# 描述目標而不是過程 (/zh-tw/docs/when-questions/describe-goal-not-process)



如果你想弄清楚如何做某事（而不是報告一個 Bug），在開頭就描述你的目標，然後才陳述重現你所卡住的特定步驟。

## 為什麼目標比步驟更重要 [#為什麼目標比步驟更重要]

經常尋求技術幫助的人在心中有個更高層次的目標，而他們在自以為能達到目標的特定道路上被卡住了，然後跑來問該怎麼走，但沒有意識到這條路本身就有問題。結果要費很大的勁才能搞定。

如果你只問某一步怎麼做，回答者只能幫你修補這一步；如果你說明真正目標，回答者可能會指出更簡單、更穩妥、更符合工具設計的方案。

## 反例：只問卡住的步驟 [#反例只問卡住的步驟]

**蠢問題：**

> 我怎樣才能從某繪圖程序的顏色選擇器中取得十六進制的 RGB 值？

## 好例子：先說明真正目標 [#好例子先說明真正目標]

**聰明問題：**

> 我正試著用替換一幅圖片的色碼（color table）成自己選定的色碼，我現在知道的唯一方法是編輯每個色碼區塊（table slot），
> 但卻無法從某繪圖程序的顏色選擇器取得十六進制的 RGB 值。

第二種提問法比較聰明，因為它讓回答者知道你真正想完成的是修改圖片色碼，而不是單純獲取某個顏色值。你可能得到像是 `建議採用另一個更合適的工具` 這樣的回覆。

## 寫法建議 [#寫法建議]

可以用這樣的順序組織問題：

1. 我最終想達到什麼結果。
2. 我目前選擇了什麼方法。
3. 我卡在這個方法的哪一步。
4. 我是否接受換一種方法。

最後一點尤其重要。很多時候，最好的答案不是把你從錯誤路徑上繼續往前推，而是幫你換一條更合適的路。
